# Deploying Probe

Probe is a 3-service app in one npm-workspaces monorepo:

| Service | What it is | Where it goes | Why |
|---|---|---|---|
| `apps/expert` | Fastify + Socket.IO realtime backend | **Render** (web service) | Needs a long-lived process for WebSocket/Socket.IO connections — not a fit for Vercel's serverless functions. |
| `apps/interviewer` | Next.js app (interviewer room + dashboards) | **Vercel** | Standard Next.js hosting. |
| `apps/interviewee` | Next.js app (candidate room) | **Vercel** | Standard Next.js hosting. |
| Database | Postgres | **Supabase** (already in use — not re-provisioned) | Both local dev and production point at the same managed Postgres; no new DB host needed. |

`packages/contract` (shared socket/event types) and `packages/db` (Prisma client) are internal workspace packages consumed by the three apps above — they don't deploy anywhere on their own.

Config files already added to the repo for this:
- `render.yaml` — Render Blueprint for `apps/expert`.
- `apps/interviewer/vercel.json`, `apps/interviewee/vercel.json` — Vercel framework hints.
- `.node-version` — pins Node 22 for both platforms.

## Before you start: what changed to make this deployable

`packages/contract` and `packages/db` used to ship raw `.ts` source with no build step — that only worked locally because `tsx`/webpack transpile on the fly. Render runs the **compiled** `apps/expert` output (`node dist/index.js`) as a plain Node process, and a bare `import ... from "@probe/contract"` resolving to a `.ts` file would only work on Node versions with (still relatively new) native TypeScript stripping. Rather than gamble on that, both packages now have a real `tsc` build step and ship compiled `dist/*.js` + `.d.ts`, so the compiled expert server runs correctly on any Node ≥ 18.

Practical side effect for local dev: **`packages/contract` and `packages/db` are no longer "live" when you edit them** — a running `dev:expert`/`dev:interviewer`/`dev:interviewee` won't pick up a change to `packages/*/src/index.ts` until you rebuild. Run this after editing either package:
```
npm run build --workspace @probe/contract
npm run build --workspace @probe/db
```
The root `postinstall` script also rebuilds `@probe/contract` automatically on every `npm install`. App code under `apps/*` is unaffected — it still hot-reloads exactly as before.

## Part 1 — Render: the `expert` backend

Deploy this first — the two Vercel apps need its URL.

### 1.1 Create the Blueprint

1. Push this repo to GitHub (Render deploys from a git repo).
2. In the Render dashboard: **New → Blueprint**, pick this repo. Render will read `render.yaml` and propose one service, `probe-expert`.
3. Before the first deploy, Render prompts for every env var marked `sync: false` in `render.yaml`. Fill them in (table below) — `AUTH_SECRET` is auto-generated for you, skip it.
4. Click **Apply**. First deploy takes a few minutes (`npm ci` installs the whole workspace, builds `@probe/contract` → `@probe/db` → `@probe/expert`).

If you'd rather create the service by hand instead of via Blueprint: **New → Web Service**, connect the repo, and set:
- **Runtime:** Node
- **Build Command:** `npm ci && npm run build:expert`
- **Start Command:** `npm run start --workspace @probe/expert`
- **Health Check Path:** `/health`
- **Plan:** at least **Starter** — the free tier spins down on idle, which drops live Socket.IO connections mid-interview. Don't use it for anything but a throwaway test.

### 1.2 Environment variables

Get the Postgres/Supabase values from the same place your local `.env` gets them (Supabase project settings → API / Database).

| Variable | Required | Value |
|---|---|---|
| `DATABASE_URL` | Yes | Supabase pooled connection string |
| `DIRECT_URL` | Yes | Supabase direct connection string |
| `NEXT_PUBLIC_SUPABASE_URL` | Recommended | Your Supabase project URL — unset keeps the insecure dev-token auth fallback (anyone can pass any user id as a bearer token) |
| `SUPABASE_SERVICE_ROLE_KEY` | Recommended | Pairs with the URL above to enable real Supabase JWT auth |
| `AUTH_SECRET` | Yes | Auto-generated by Render's Blueprint. If you create the service by hand, set this to a real random string — the code defaults to an insecure placeholder if unset. |
| `FRONTEND_URL` | Yes | Interviewee's Vercel URL (set in Part 3, after Vercel exists) |
| `COMPANY_FRONTEND_URL` | Yes | Interviewer's Vercel URL (set in Part 3) |
| `NEXT_PUBLIC_APP_URL` | Optional | Same as `FRONTEND_URL` — extra CORS allowlist entry |
| `NEXT_PUBLIC_COMPANY_APP_URL` | Optional | Same as `COMPANY_FRONTEND_URL` — extra CORS allowlist entry |
| `JUDGE0_API_URL` / `JUDGE0_API_KEY` | Optional | Code execution ("Run"/"Submit") is disabled without these. RapidAPI Judge0 CE works out of the box with `JUDGE0_PROVIDER=auto`. |
| `GOOGLE_GENERATIVE_AI_API_KEY` / `XAI_API_KEY` / `GROQ_API_KEY` | Optional | Any one enables the Probe copilot (rubric/suggestions/scorecard). None set → deterministic fallback rubric, no AI suggestions. |
| `MONGODB_URI` | Optional | Extra question bank source. Unset → falls back to the seeded Postgres bank. |

`EXPERT_HOST` and `NODE_ENV` are already set by `render.yaml`. Port binding is automatic — `apps/expert/src/lib/env.ts` now prefers Render's injected `PORT` over the local-dev `EXPERT_PORT`.

You won't have real values for `FRONTEND_URL`/`COMPANY_FRONTEND_URL` yet on the first deploy — put in placeholders (e.g. `https://placeholder.vercel.app`) and come back in Part 3 once the Vercel URLs exist. The service works fine in the meantime; only cross-origin browser requests from the not-yet-correct origins would fail until you fix it.

### 1.3 One-time database setup

If `DATABASE_URL`/`DIRECT_URL` point at the **same Supabase project you already use for local dev**, the schema is already pushed and seeded — skip this. If you're pointing at a fresh Supabase project instead, run once from your machine with `packages/db/.env` pointed at that project:
```
npm run db:generate
npm run db:push
npm run db:seed
```

### 1.4 Note the deployed URL

Render gives you something like `https://probe-expert.onrender.com`. You need this for both Vercel projects in Part 2.

## Part 2 — Vercel: `interviewer` and `interviewee`

These are two **separate Vercel projects** from the same repo.

### 2.1 Import the interviewer app

1. Vercel dashboard → **Add New → Project** → import this repo.
2. **Root Directory:** `apps/interviewer` (click "Edit" next to Root Directory during import — this is a manual step, it can't be set from a committed file).
3. Framework is auto-detected as Next.js (confirmed by `apps/interviewer/vercel.json`). Leave Build/Install commands on their defaults — Vercel runs `npm install` at the detected workspace root (this repo root), which also triggers the `postinstall` script that builds `@probe/contract`, then runs `next build` inside `apps/interviewer`.
4. If prompted, make sure **"Include files outside the Root Directory"** is enabled — the app depends on `packages/contract` via a workspace symlink outside `apps/interviewer`. Vercel enables this automatically for detected npm-workspace monorepos; only check it manually if the build fails with `Module not found: Can't resolve '@probe/contract'`.
5. Environment variables (Production — repeat for Preview if you want preview deploys working too):

   | Variable | Value |
   |---|---|
   | `NEXT_PUBLIC_EXPERT_URL` | Your Render URL, e.g. `https://probe-expert.onrender.com` |
   | `NEXT_PUBLIC_CANDIDATE_URL` | The interviewee Vercel URL (from step 2.2 below — come back and set this after) |
   | `NEXT_PUBLIC_ICE_SERVERS` | `[{"urls":"stun:stun.l.google.com:19302"}]` (default STUN; add a TURN server here if cross-network video needs it) |

6. Deploy.

### 2.2 Import the interviewee app

Same steps, a second Vercel project:
1. **Root Directory:** `apps/interviewee`.
2. Env vars:

   | Variable | Value |
   |---|---|
   | `NEXT_PUBLIC_EXPERT_URL` | Same Render URL as above |
   | `NEXT_PUBLIC_ICE_SERVERS` | Same as above |

3. Deploy.

### 2.3 Cross-wire the two Vercel URLs

Once both are deployed you have two URLs, e.g.:
- `https://probe-interviewer.vercel.app`
- `https://probe-interviewee.vercel.app`

Go back to the interviewer project's env vars and set `NEXT_PUBLIC_CANDIDATE_URL` to the interviewee URL, then redeploy the interviewer project (env var changes need a redeploy to take effect).

## Part 3 — Close the loop: point Render's CORS at the real Vercel URLs

Back in the Render dashboard, update `probe-expert`'s env vars:

| Variable | Value |
|---|---|
| `FRONTEND_URL` | `https://probe-interviewee.vercel.app` |
| `COMPANY_FRONTEND_URL` | `https://probe-interviewer.vercel.app` |
| `NEXT_PUBLIC_APP_URL` | same as `FRONTEND_URL` |
| `NEXT_PUBLIC_COMPANY_APP_URL` | same as `COMPANY_FRONTEND_URL` |

Save — Render redeploys automatically on env var changes. This is what makes `@fastify/cors` (REST) and Socket.IO's own CORS config accept requests from your real domains.

---

## Verify it end to end

1. `https://probe-expert.onrender.com/health` → `{"ok":true,"service":"expert"}`.
2. Open the interviewer URL, sign in (see Auth note below), create an instant interview.
3. Open the interviewee URL with the candidate join link from step 2.
4. Confirm: lobby → admit → video/screen-share/shared editor/Run all connect, exactly like the local 3-terminal setup.

## Known limitations carried over from the current build

- **Auth is dev-token unless you set the Supabase env vars.** Without `NEXT_PUBLIC_SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` on Render, the expert service accepts any string as a raw user id — fine for a demo, not for a real public deployment. This was already flagged as unfinished in the project's own status notes (real Supabase login was never wired into the frontends), not something introduced by this deploy.
- **Resume uploads are ephemeral on Render.** They're written to a local `uploads/` folder (`apps/expert/src/index.ts`), which does not persist across redeploys/restarts on Render's default filesystem. Attach a Render Disk mounted at `apps/expert/uploads` if this needs to survive, or move it to object storage later.
- **Preview deployments won't have working CORS.** Vercel preview URLs are random per-deploy; the CORS allowlist above only covers the two production URLs. Fine for production, expect preview deploys of the room to fail CORS until you either add their origin manually or accept previews are for UI-only checks.
- **Free-tier Render spins down on idle** — don't use it for this app; an in-progress interview would drop its socket connection mid-session on a cold start.
- **Cross-network video may need a TURN server.** The default `NEXT_PUBLIC_ICE_SERVERS` is Google's public STUN only, which fails behind some NATs/corporate firewalls. Add a TURN server to that JSON array on both Vercel projects if you hit connection issues between real remote participants.
